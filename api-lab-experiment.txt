1. Executive Summary
The Moodle-Integrated Online Exam System is a full-stack, multi-role web platform engineered to streamline
online assessment creation, dynamic student examination, automated real-time evaluation, and synchronous
user/course synchronization with external Learning Management Systems (LMS), specifically Moodle. Built on
the robust MERN stack (MongoDB, Express.js, React 19, Node.js), the platform guarantees high scalability,
decoupled microservice-ready modularity, and low latency during concurrent high-stakes exam submissions.
This report presents a thorough breakdown of the platform's multi-tiered architectural layout, an exhaustive
specification of all internal REST APIs and external Moodle integration endpoints, and a rigorous technical
justification for every component and technology choice made.
2. System Architecture & Component Topology
The system follows a modernized Client-Server-Database-External API architectural pattern. By enforcing
strict physical and logical separation between the user presentation layer, central business logic processing,
persistent data storage, and third-party enterprise services, the application achieves fault tolerance, seamless
maintainability, and independent scaling capabilities.

Detailed Layer Breakdown:
1. Frontend Presentation Layer (React 19 + Vite): Serves as the interactive single-page application (SPA). It
 dynamically loads role-specific views (Admin, Instructor, Student), manages live client-side countdown
timers, handles interactive exam rendering, and communicates asynchronously via Axios HTTP requests.
2. Backend Application Layer (Node.js + Express.js): Encapsulates all domain business logic. It exposes
 secure RESTful endpoints, executes JWT authentication and Role-Based Access Control (RBAC), verifies
submission deadlines on the server side, automatically computes grades upon submission, and formats
integration payloads for Moodle.
3. Data Persistence Layer (MongoDB + Mongoose): Stores system state documentations across three
 primary collections: Users, Exams (with embedded polymorphism for questions), and Attempts. Mongoose
provides object-data modeling (ODM), enforcing type definitions, validation, and automated indexing.
4. Enterprise Integration Layer (Moodle Web Services API): Wraps external HTTP calls to Moodle's REST
 endpoints (`server.php`). It seamlessly syncs student rosters, instructor assignments, and course offerings
directly from an institutional Moodle installation.

4.2 External Moodle Web Services Integration Client
The file backend/moodle_api/client.js implements a dedicated wrapper class, MoodleAPIClient, which
encapsulates calls to Moodle's REST server endpoint (/webservice/rest/server.php). Moodle requires
parameters encoded as application/x-www-form-urlencoded with complex array syntax. The client automatically
serializes JavaScript arrays into standard Moodle URL notation (e.g. userids[0]=2).

6. Security & Operational Workflows
Key Security & Integrity Mechanisms:
1. Server-Side Timer Enforcement: Client-side timers in React are prone to tampering. When a student
 starts an exam, the backend locks `start_time` in MongoDB. During answer submissions or final submission,
the backend independently verifies that elapsed time does not exceed `duration + grace_period`.
2. Automated Server-Side Grading: Scores are calculated strictly inside `studentController.submitExam`.
 Answer keys are never transmitted to the frontend during an ongoing exam, preventing client-side
inspection or reverse-engineering.
3. Role-Based Authorization Middleware: Express endpoints are shielded by `protect` and
 `authorize('admin', 'instructor')` middlewares, inspecting JWT signature and roles before granting access to
critical operations.
